  已验证的关键发现

  1. 前台加速是核心 — CF 验证从 90s → 15s（hasFocus=True 时快 4-6x）
  2. Popup → JS fetch 可行 — SD 的 pdfft 端点绕不开，但 popup 内 re-fetch 能拿到 PDF
  3. 临时 profile + CF 信任积累 — 比反复杀真实 Edge 更稳定
  4. 真实 profile 有 session 恢复问题 — 杀/启多次后状态不可靠

  当前的问题

  1. 可移植性差 — Edge 路径硬编码、Windows API 绑定
  2. Publisher 适配分散 — SD 的逻辑写在具体方法里，扩展其他网站要复制粘贴
  3. Profile 管理混乱 — 临时 vs 真实各有坑，没统一策略
  4. 错误恢复弱 — CF 卡住、SPDF-3 权限错误等处理不够系统

                      ┌─────────────────────┐
                      │   fetch_paper(doi)   │  ← 统一入口
                      └─────────┬───────────┘
                                │
                      ┌─────────▼───────────┐
                      │   Strategy Router   │  ← 按 DOI/URL 选策略
                      └─────────┬───────────┘
                                │
             ┌──────────────────┼──────────────────┐
             ▼                  ▼                  ▼
      ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
      │ Direct HTTP  │  │  CDP Browser │  │  HTML Text   │
      │ (curl_cffi)  │  │ (Playwright) │  │ (BeautifulSoup)│
      └──────┬───────┘  └──────┬───────┘  └──────┬───────┘
             │                  │                  │
             └──────────────────┼──────────────────┘
                                ▼
                      ┌─────────────────────┐
                      │   Publisher Adapter │  ← 每站一个适配器
                      │  SD / arXiv / Nature │
                      │  Wiley / Springer /..│
                      └─────────────────────┘

  核心思路

  1. Strategy Router — 根据 URL/DOI 自动选策略
  - 10.48550/ → Direct HTTP（arXiv 不需要浏览器）
  - 10.1016/ → CDP Browser（SD 必须走浏览器）
  - 10.1038/ → 先 HTTP 试，失败切 CDP

  2. Publisher Adapter — 每个网站只要写 3 个东西
  - detect(paper) → bool — 是不是这个网站
  - extract_pdf_link(page) → str — 怎么找 PDF 链接
  - extract_text(html) → str — 怎么提取正文

  3. Browser Session 管理 — 统一处理 profile
  - 独立 profile 目录（不碰用户真实浏览器）
  - 首次使用引导用户手动登录一次 CARSI
  - 后续自动复用，CF 信任积累

  4. 跨平台 — Chrome 替代 Edge
  - macOS/Linux 用户没有 Edge
  - Chrome/Chromium 的 CDP 完全一样
  - 自动检测系统中可用的浏览器
